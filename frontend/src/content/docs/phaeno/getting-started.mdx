# Phaeno operations guide

The Phaeno organization workspace is the operating surface for organization administration, data provisioning, Customer and Partner order work, laboratory execution, public Website intake, commercial configuration, and integration recovery.

## Navigate workspaces and guides

The POMS dashboard, Lab operations, Order operations, Data provisioning, and Documentation use a shared sidebar anchored beneath the toolbar at the far-left edge of the screen. Its header identifies the current workspace and, on wider screens, provides the pin or unpin icon. The sidebar is pinned by default, leaves a centered page in place when the left margin has room, and reflows the page only to prevent overlap. Pointer users can preview an unpinned sidebar by approaching the edge; keyboard and touch users open the same non-modal rail from the tab.

Documentation automatically lists only the guides for the current organization, with an icon identifying each topic. Data provisioning, Order operations, and Laboratory operations expand to show an overview and detailed workflow guides. The group containing the current guide opens automatically; use its disclosure button to expand or collapse it.

## Work in the Phaeno context

Phaeno users enter the Phaeno organization context through their authenticated session. The user menu does not provide an act-as control for switching into Prospect, Customer, or Partner organizations. Manage those organizations and their access through the **Organizations** workspace.

Platform capabilities come from an active administrator membership in the active Phaeno organization. Authentication-provider roles are not used for portal authorization.

## Find platform work

- **Dashboard** uses the sidebar to show one Order Operations, Lab Operations, Accounts, or Web Operations panel at a time. Counts summarize the work or intake represented by each panel.
- **Accounts** opens the durable external-account directory and workspace. The internal Phaeno organization is not listed. Normal onboarding begins in HubSpot, so the standard Accounts screen does not create an account or manually submit a request. Until HubSpot intake is connected, the page identifies that integration state while existing account details, Portal requests, service entitlements, members, and invitations remain available through the connected API.
- **Web Operations** is a Phaeno-administrator view of active public Website intake. Use the **Mailing List** and **Demo Requests** buttons to select one list panel at a time. Mailing List shows the newest active signups and identifies technical-brief requests; use **Unsubscribe** and confirm to remove a contact from that active list. Demo Requests shows the inquiry contact, organization, email, and submitted description; use **Mark complete** and confirm after the inquiry has been handled. Completed or unsubscribed items leave their active list immediately, but the original Website intake and action audit remain in POMS. Each panel paginates independently in groups of 10; its footer paginator appears only when the list has more than one page. These actions do not create an Account, Portal request, HubSpot contact, or operational order.
- **User management** remains a session-only preview. Do not rely on additions, role changes, deactivations, or reactivations made on that standalone screen.
- **Data provisioning** manages source samples, curated packages, grants, and governance.
- **Order operations** contains laboratory order intake, Customer lab, Partner PSeq kit, Partner assembly, and integration-recovery queues.
- **Lab operations** contains Phaeno-only work orders for receipt, accession, protocol execution, materials and equipment, libraries and batches, outsourced sequencing, exceptions, and scientific approval. Access requires an effective Lab role or platform-administrator bootstrap access.
- **Order configuration** contains commercial, scientific, Partner-offering, credit, and integration settings.
- **Docs** opens the guide set for the current organization.

## Separate safe and internal information

Tenant timelines and messages must contain only tenant-safe reasons and actions. Internal notes, scientific investigation evidence, credentials, retry details, and other restricted facts stay in authorized Phaeno surfaces.

Never place passwords, OAuth credentials, webhook secrets, connection strings, protected health information, or customer file contents in notes, audit descriptions, notification messages, or QuickBooks memo fields.

## Support users safely

Confirm the selected organization, user membership, capability, record reference, and visible status before changing data. Use the Prospect, Customer, or Partner documentation view to understand what the caller can see. Do not ask users to share credentials or authentication tokens.
